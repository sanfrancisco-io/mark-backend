# AI Workflow — marketplace-backend

Документ описывает как AI-ассистент (Claude) использовался при разработке backend.
Цель — воспроизвести подход в следующих сессиях и оценить эффективность.

---

## Инструменты

### Claude Code (claude-sonnet-4-6)
Основной инструмент. Запускается в IDE (VSCode extension) с доступом к файловой системе проекта.
Умеет читать, создавать и редактировать файлы, выполнять bash-команды, искать по коду.

### Контекстные файлы
Два файла поддерживаются в актуальном состоянии и загружаются в начале каждой сессии:

| Файл | Назначение |
|---|---|
| `backend_todo.md` | Полный список задач по фазам. Чекбоксы отмечаются по мере выполнения. |
| `todo-state.md` | Снимок текущего состояния: последняя выполненная задача, следующая, принятые решения, лог сессий. |

**Протокол начала сессии:** прочитать оба файла → определить текущую фазу → предложить план → ждать подтверждения.

### Сессионные логи
`docs/ai/sessions/session_NN_name.md` — итоги каждой сессии: что делали, сколько итераций, какие проблемы, почему приняли те или иные решения.

---

## Общий подход

### Plan → Confirm → Execute
Каждая фаза начинается с предложения плана в текстовом виде.
Код пишется только после явного подтверждения пользователя.
Это позволяет скорректировать подход до написания кода, а не переписывать уже написанное.

### Одна фаза — одна итерация
Цель: реализовывать каждую фазу за один проход без переделок.
Достигается за счёт:
- уточнения требований в рамках обсуждения плана
- явных вопросов по неоднозначным местам перед стартом

### Вопросы перед кодом
Если в плане есть развилка — задаём вопрос до начала реализации.
Примеры из этого проекта:
- "total или has_more в пагинации?" → has_more (infinite scroll)
- "дублировать ensure_bucket_exists в Python или убрать?" → оставить как страховку
- "seed в автостарте compose или только вручную?" → только вручную

### Обновление состояния после каждой фазы
После проверки каждой фазы пользователем:
1. Отмечаем чекбоксы в `backend_todo.md`
2. Обновляем `todo-state.md` (последняя задача, следующая, решения)
3. При необходимости — пишем сессионный лог

---

## Что работало хорошо

- **Явное подтверждение плана** — ни разу не пришлось переписывать код из-за неверно понятых требований
- **todo-state.md как "память" между сессиями** — не нужно пересматривать историю чата чтобы понять где остановились
- **Вопросы по развилкам** — пользователь уточнял требования (has_more, stock в Product, seed вручную) до того как код был написан

## Что можно улучшить

- Автоматическая проверка (тесты) сразу после реализации фазы — сейчас проверка ручная
- `PROMPTS.md` с шаблонами промптов для типовых операций (начало сессии, создание роутера, написание теста)
